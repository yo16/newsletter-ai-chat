# /batch-change

問題修正・改善・変更のための管理コマンド。
問題報告や改善要望に対して、原因分析→対応方針の提示→ユーザー承認→Beads登録→実装という規律あるプロセスで処理する。

## 目的
- コード修正だけでなく、**問題の背景・原因・発覚のきっかけを記録する**
- 変更を Beads タスクとして管理し、トレーサビリティを確保する
- ユーザーの承認なしに修正を実行しない

## 引数
- `$ARGUMENTS` : 問題・改善要望の内容（任意。change-gate スキルから自動的に引き渡される場合もある）

## 使用するサブエージェント
- `beads-manager` : Beadsタスクの登録・ステータス管理
- `coder` : コード実装（承認後）
- `db-coder` : DB操作関数の実装（承認後、DB関連の場合）
- `tester` : テスト実行（実装後）
- `git-manager` : ブランチ・コミット操作

## 処理フロー

### 1. 問題・要望の整理（Record）
- ユーザーの報告内容を以下の構造で整理する:
  - **現象**: 何が起きているのか（エラーメッセージ、期待と実際の差分）
  - **発覚のきっかけ**: どういう操作・状況で気づいたのか
  - **対象**: どの画面・機能・コンポーネントに関するものか
  - **種別**: バグ修正 / 機能改善 / リファクタリング / 仕様変更
- 不明な点があればユーザーに確認する

### 2. 原因分析（Analyze）
- 関連するコードを読み、問題の原因を調査する
- 参照すべきドキュメント:
  - `docs/design.md` : アプリケーション設計（影響範囲の把握）
  - `docs/db-design.md` : DB設計（データ関連の問題の場合）
  - `docs/specification.md` : 仕様との乖離の確認
- 分析結果を以下の構造で整理する:
  - **根本原因**: なぜこの問題が発生しているのか
  - **影響範囲**: この問題が影響している箇所の一覧
  - **設計書の更新要否**: `docs/design.md` や `docs/db-design.md` の修正が必要か

### 3. 対応方針の提示（Propose） → ユーザー承認
- 修正方針を提示する:
  - **修正内容**: 何をどう変更するか（具体的なファイル・箇所）
  - **修正量の見積もり**: 変更するファイル数・行数の概算
  - **リスク**: 修正に伴うリスクや副作用
  - **設計書の更新内容**: 更新が必要な場合、その内容
  - **代替案**: 他のアプローチがある場合は併記する
- **ユーザーの承認を待つ（承認なしに次のステップに進まない）**

### 4. Beadsタスクの登録（Register） → `beads-manager`
- ユーザーの承認を得たら、Beadsタスクとして登録する
- タスクの description には以下を必ず含める:

```markdown
## 背景
- 発覚のきっかけ: （ユーザーの報告内容）
- 現象: （何が起きていたか）

## 原因
- （根本原因の説明）

## 対応方針
- （承認された修正内容）

## 変更対象ファイル
- （ファイル一覧）

## テスト観点
- （修正に対するテストケース）
```

- 既存の依存関係がある場合は設定する

### 5. 実装（Implement） → `coder` / `db-coder`
- `/batch-task-execute` と同じプロセスで実装する:
  - `feature/t-xxx` ブランチを作成
  - コーディング（coder / db-coder）
  - テスト実行（tester）
  - テストレビュー（test-reviewer）によるテスト十分性の評価
  - テスト成功かつレビュー合格後、`dev` へマージ
- 設計書の更新が必要な場合は、コード修正と合わせて更新する

### 6. 完了報告（Report）
- 以下をユーザーに報告する:
  - 修正内容の概要
  - 変更したファイルの一覧
  - テスト結果
  - 設計書を更新した場合はその内容
  - Beadsタスクのステータス（完了）

## 重要な制約
- **ステップ3（対応方針の提示）でユーザーの承認を得るまで、コードの修正を一切行わない**
- **ステップ4のBeads登録を省略しない**（どんなに軽微な修正でも登録する）
- Beadsタスクの description には「背景」「原因」セクションを必ず含める
- 失敗時は `/batch-failed` の処理に従う
