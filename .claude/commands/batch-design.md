# /batch-design

設計フェーズのコマンド。
要件定義書（`docs/specification.md`）を入力として、開発者が実装を開始できるレベルのソフトウェア設計書を作成する。
設計結果はアプリケーション設計を `docs/design.md` に、DB設計を `docs/db-design.md` に出力する。

## 設計フェーズのゴール
「設計書を読めば、開発者がアーキテクチャ上の重要な判断を自ら行う必要なく、実装に着手できる」状態を作ること。

## 引数
- `$ARGUMENTS` : 設計対象の補足説明（任意）

## 使用するサブエージェント
- `app-designer` : システムアーキテクチャ・アプリケーション設計の主担当
- `db-designer` : DB設計・API設計の主担当
- `supabase-specialist` : Supabase利用時の固有機能に関する助言・設計

## 処理フロー

### 1. 要件の理解（Analyze）
- `docs/specification.md` を読み込み、アプリケーション仕様を理解する
- `CLAUDE.md` を読み込み、プロジェクトルールを理解する
  - 特に「DBサービス」「DB操作アーキテクチャ」「API技術スタック」「テスト方針」「環境変数」セクションを確認する
  - これらが未記載の場合は、ステップ2で合わせてユーザーに確認し、設計完了後に `CLAUDE.md` への追記を案内する
- $ARGUMENTS が指定されている場合は、その内容も考慮する
- 以下の観点で仕様を分析する:
  - **機能要件**: アプリケーションの主要機能、ユーザーが行う操作
  - **非機能要件**: パフォーマンス、セキュリティ、可用性、スケーラビリティ
  - **データ要件**: 管理するデータエンティティ、リレーション、データ量の増加傾向
  - **ユーザーフロー**: 画面数、画面遷移、主要なユーザー操作シナリオ
  - **外部連携**: 外部API・サービスとの接続要件
  - **制約**: 技術的制約、ビジネス上の制約

### 2. 対話的な要件探索（Explore）
- 仕様書だけでは不明確な点がある場合、ユーザーに確認する
- 確認すべき観点（`CLAUDE.md` に記載済みの項目はスキップする）:
  - **技術スタック**:
    - 使用するDBサービス（Supabase / その他）
    - API技術スタック（Next.js API Routes / Server Actions / Route Handlers 等）
    - ORM / DBクライアント（Prisma / Drizzle / Supabase Client 等）
    - スタイリング手法（CSS Modules / Tailwind CSS 等）
    - 認証ライブラリ（NextAuth / Clerk / Supabase Auth 等、必要な場合）
  - **アーキテクチャ方針**:
    - DB操作関数の配置場所（デフォルト: `src/lib/db/`）
    - 状態管理の方針（React標準 / 外部ライブラリ）
    - テスト方針（モック使用 / テスト用DB接続）
  - **運用・データ管理**:
    - データの保持期間・削除ポリシー
    - 想定されるデータ量・アクセス頻度
    - 既存データの移行要否
    - バックアップ要件
- ユーザーの回答を踏まえて設計方針を決定する

### 3. システムアーキテクチャ設計（Architecture Design） → `app-designer`
要件分析の結果を踏まえ、システム全体のアーキテクチャを設計する。

#### 3-1. 技術スタックの確定と根拠
- 使用する技術スタックを一覧化し、各選定理由を明記する
- 主要ライブラリのバージョン方針を記載する

#### 3-2. システム構成
- システム全体のレイヤー構成を設計する（プレゼンテーション層 / アプリケーション層 / データアクセス層）
- 各レイヤーの責務と依存関係の方向を定義する
- 外部サービスとの接続ポイントを明示する

#### 3-3. 画面設計
- 画面一覧と各画面の役割・主要機能を定義する
- 画面遷移図（テキスト表現）を作成する
- 各画面の主要な構成要素を特定する

#### 3-4. ルーティング設計
- App Routerに基づくルート一覧と対応する画面
- レイアウト共有戦略（layout.tsx の配置）
- 動的ルート・ルートグループの設計
- ミドルウェアの配置と役割

#### 3-5. レンダリング戦略
- ルートごとのレンダリング方式を決定する（SSR / SSG / ISR / CSR）
- 選定理由を明記する（データの更新頻度、パーソナライズ要否、SEO要件等）
- Streaming / Suspense 境界の設計（必要な場合）

#### 3-6. コンポーネント設計
- 共通コンポーネントの抽出と責務定義
- ページ固有コンポーネントの構成
- Server Components / Client Components の境界と判断理由
- コンポーネント間のデータの流れ

#### 3-7. データフロー設計
- データの取得・更新がシステム内をどう流れるかを設計する
- Server Components でのデータフェッチ方式
- Server Actions / Route Handlers の使い分け
- キャッシュ戦略（revalidate の設定方針、キャッシュの無効化タイミング）
- 楽観的更新の適用箇所（必要な場合）

#### 3-8. 状態管理設計
- サーバー状態とクライアント状態の分離方針
- フォーム状態の管理方針
- URL状態の活用（検索パラメータ、フィルタ等）
- グローバル状態が必要な箇所の特定と管理方式

#### 3-9. 認証・認可設計（要件にある場合）
- 認証フロー（ログイン / ログアウト / セッション管理）
- 認可モデル（ロール定義、権限マトリクス）
- ミドルウェアによるルート保護の設計
- 認証状態のクライアントへの伝播方式

#### 3-10. エラーハンドリング設計
- エラーの分類（バリデーションエラー / ビジネスロジックエラー / システムエラー）
- 各層でのエラー処理方針
- error.tsx / not-found.tsx の配置戦略
- ユーザーへのエラー表示方針（トースト、インライン、ページ遷移等）
- バリデーション戦略（サーバー側 / クライアント側の役割分担、バリデーションライブラリ）

#### 3-11. ディレクトリ構成
- 完全なディレクトリツリーの設計
- 各ディレクトリの責務と配置ルール
- ファイル命名規約

#### 3-12. アプリケーションが必要とするデータ操作の洗い出し
- 各画面・機能が必要とするCRUD操作を一覧化する
- **この一覧をdb-designerに引き渡し、DB設計・API設計のインプットとする**

### 4. DB設計（DB Design） → `db-designer`
- app-designerが洗い出したデータ操作要件も踏まえて設計する
- テーブル設計を行う:
  - エンティティの抽出と定義
  - カラム定義（型、制約、デフォルト値）
  - リレーション設計（外部キー、中間テーブル）
  - 正規化レベルの判断
  - インデックス設計
- DB操作API設計を行う:
  - エンドポイント / Server Actions / DB操作関数の一覧
  - 各操作のインターフェース仕様（入力パラメータ、戻り値の型）
  - エラーハンドリング方針
  - ページネーション・フィルタリング仕様

### 5. Supabase固有の設計（Supabase利用時） → `supabase-specialist`
- 使用するDBサービスがSupabaseの場合にのみ実行する
- `db-designer` の設計をベースに、以下を検討:
  - RLS（Row Level Security）ポリシーの設計
  - Supabase Auth連携の設計
  - Storage利用の設計（必要な場合）
  - Realtime機能の適用判断（必要な場合）
  - Edge Functions活用の検討（必要な場合）
  - Supabase固有の機能による設計の最適化提案
- `db-designer` の設計に対するフィードバックを提供し、必要に応じて設計を修正する

### 6. 設計の統合と検証（Validate）

#### 6-1. アプリケーション設計とDB設計の整合性
- アプリケーションが必要とするデータ操作がすべてAPI/関数として定義されているか
- データフロー設計とAPI設計に矛盾がないか
- コンポーネントが受け取るデータの型とAPIのレスポンス型が整合しているか
- レンダリング戦略とデータフェッチ方式が矛盾しないか

#### 6-2. 非機能要件の充足
- **パフォーマンス**: レンダリング戦略・キャッシュ戦略・インデックス設計は要件を満たすか
- **セキュリティ**: 認証・認可・入力検証・データ保護は適切か
- **保守性**: 責務分離は適切か、ディレクトリ構成は明確か、命名規約は一貫しているか
- **テスタビリティ**: 設計がテスト可能な構造になっているか

#### 6-3. 設計上のリスクと対策
- 技術的リスク（未経験の技術、複雑な要件）を特定する
- 各リスクに対する緩和策を記載する
- 設計上のトレードオフを明示する（例: パフォーマンスとシンプルさ、正規化と読み取り速度）

- 問題があれば設計を修正する

### 7. 設計書の出力（Document）
- アプリケーション設計を `docs/design.md` に出力する
  - フォーマットは `app-designer` エージェントの「成果物」セクションに従う
- DB設計・API設計を `docs/db-design.md` に出力する
  - フォーマットは `db-designer` エージェントの「成果物」セクションに従う
- `docs/design.md` の冒頭にDB設計書（`docs/db-design.md`）への参照リンクを記載する

### 8. 環境変数の定義
- `.env.sample` に必要な環境変数を記載する
- DB接続情報、APIキー等の環境変数名を定義する（値は空にする）
- Supabase利用時は、Supabase固有の環境変数も含める

### 9. ユーザーへの報告
- 設計の概要をユーザーに報告する:
  - システム構成の概要（レイヤー構成、主要技術）
  - 画面数、主要な画面遷移フロー
  - レンダリング戦略の概要
  - テーブル数、主要エンティティ
  - APIエンドポイント / Server Actions の数
  - セキュリティ方針の概要
  - 設計上のリスク・トレードオフの説明
- `docs/design.md` と `docs/db-design.md` の確認を依頼する
- `CLAUDE.md` にDB関連セクションが未記載の場合、設計で決定した内容に基づく追記案を提示する
- 設計の承認後、タスク分解（task-decomposer）に進むよう案内する
