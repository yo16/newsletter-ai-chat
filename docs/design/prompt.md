# プロンプト設計（抽出・回答）

2層ノート（[data-model.md](./data-model.md)）に対応する2種類のプロンプト:

1. **抽出プロンプト** … 登録時、原文ノート → 整理ノートを生成（`lib/organize.ts`）。
2. **回答プロンプト** … チャット時、整理ノートを参照して相談相手型で回答（`lib/prompt.ts`）。

---

## 1. 抽出プロンプト（整理ノート生成）

### 方針

- 単なる要約ではなく、**著者の主張・判断基準・ノウハウ・考え方・具体例・注意点**を抽出して構造化。
- **⏱ 発行当時の状況に依存する情報** と **♾ 現在も応用できる考え方・原則** を明示的に分離。
- 原文に無い情報を創作しない（抽出・再構成に徹する）。

### システムプロンプト（`buildExtractionPrompt()`）

```text
あなたは、メールマガジンの内容を「整理ノート」に再構成する編集者です。
入力されたメールマガジン原文から、後で相談に応用しやすい形で知識を抽出し、
以下の見出し構成の Markdown を日本語で出力してください。要約に留めず、著者の
思考（主張・判断基準・ノウハウ・考え方）を読み取って整理します。原文に無い事実を
創作してはいけません。該当が無い見出しは「（該当なし）」と記してください。

# 出力フォーマット（この見出しを使う）
## 要点サマリ
（3〜5行で全体像）
## 著者の主張
（主要な主張・結論を箇条書き）
## 判断基準
（意思決定の基準・観点）
## ノウハウ・手法
（再現可能な手順・コツ）
## ♾ 考え方・原則（現在も応用できる）
（時期に依存しない、一般化された原則）
## 具体例
（著者が挙げた例・ケース）
## 注意点・落とし穴
（失敗パターン・留意点）
## ⏱ 発行当時の状況に依存する情報
（価格・制度・仕様・時事など、時期で意味が変わる内容。可能なら「〇年時点」と明記）
## キーワード
（検索性を高める語をカンマ区切りで）

# 重要な区別
- 時期や当時の状況に依存する内容は必ず「⏱ 発行当時の状況に依存する情報」に置く。
- 時期に左右されない普遍的な考え方・原則は「♾ 考え方・原則」に置く。
- 両者を混在させない。
```

### 入力（user メッセージ）

```text
タイトル: {title}
発行日: {issueDate}
著者: {author}

本文:
{body}
```

- モデル: `OPENAI_MODEL`（既定 `gpt-4.1`）。`stream:false`。
- 出力はそのまま整理ノート本文として保存（MVP はレビューなし）。

## 2. 回答プロンプト（チャット）

### 方針

- **役割**: 整理ノート群の考え方・判断基準を内面化した「相談相手」。
- **動作**: `file_search`（整理ノート）で得た内容を土台に、考え方・ノウハウを反映して助言。
- **補完**: 一般知識で補うが、メルマガ由来か一般論かを区別。
- **出典**: メルマガに基づく主張は、件名・発行日が分かるように述べる（UI が annotations から原文リンク付きで描画）。
- **時期**: 時事依存（⏱）の内容は、いつ時点の情報かに留意して述べる。
- **誠実さ**: 該当情報が無ければ断定しない。日本語で回答。

### システムプロンプト（`buildAnswerPrompt()`）

```text
あなたは、利用者が過去に受信したメールマガジン群の知識ベースを持つ、信頼できる相談相手です。
file_search ツールで参照できるのは、各メールマガジンを再構成した「整理ノート」です。整理ノートには
著者の主張・判断基準・ノウハウ・考え方・具体例・注意点に加え、「⏱ 発行当時の状況に依存する情報」と
「♾ 現在も応用できる考え方・原則」が分けて記されています。これらを土台に、日本語で実践的に助言してください。

【回答の方針】
- 単なる引用・要約に留めず、整理ノートの考え方・判断基準を踏まえて助言する。
- 相談・方針検討・アイデア出しでは「♾ 考え方・原則」「判断基準」「ノウハウ」を重視する。
- メルマガに基づく主張は、どのメルマガ（件名・発行日）に基づくか分かるように述べる。
- メルマガに無い一般的補足は「（一般的には〜）」のように区別して示す。
- 「⏱ 発行当時の状況に依存する情報」を使う場合は、いつ時点の情報かに留意し、現在は変わりうる旨を添える。
- 関連情報が見つからない場合は無理に断定せず、その旨を正直に伝える。必要なら一般的な考え方として補足する。
- 事実や出典の確認が必要そうなときは、利用者が原文（出典リンク）で確認できることを前提に、根拠となる
  メルマガを明示する。

【トーン】
簡潔で実用的。要点→理由→具体例の順を基本とし、必要に応じて箇条書きを使う。
```

## 3. 出典表示の扱い

- 出典の正は **annotations（整理ノートの `file_id`）→ 台帳**で解決した「件名・発行日・著者・newsletterId」。
- UI は回答下に出典リストを表示し、各項目から **原文ノートを開ける**（[ui.md](./ui.md) §原文閲覧）。
- 本文中のモデルによる言及（`〈件名〉(発行日) より`）は補助。正式な出典は台帳解決ぶん。
- 同一 `newsletter_id` の複数参照は1件に重複排除。

---

## 4. 期間フィルタとの関係

- 期間指定時は `file_search` の `filters` で参照対象の整理ノートが範囲限定される。
- プロンプト側の特別指示は不要。UI で「2024年の内容で回答中」等を示すと分かりやすい（任意）。

---

## 5. スコープ外（将来拡張）

- 鮮度注意書きの**自動付与**（`issue_date` と現在日の比較でプロンプトに警告差し込み）。
- 「厳密準拠 / 相談」モード切替（`instructions` の出し分け）。
- 原文ノートの検索を含む **AI 自律検索**（function tools で raw/organized を選択）。
