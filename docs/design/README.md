## 主要な設計判断（ADR ライト）

| # | 論点 | 決定 | 理由 |
|---|---|---|---|
| D1 | OpenAI API 面 | **Responses API + `file_search`** | Assistants API は 2026-08-26 終了。後継かつ `filters` を直接渡せる。 |
| D2 | 期間絞り込み | 発行日を **`issue_date`（Unix 秒）属性**で登録し `gte`/`lte` フィルタ | 属性の数値比較で日付レンジを表現。 |
| D3 | 知識モデル | **2層**：原文ノート（忠実保存）＋ 整理ノート（抽出・整理） | 事実確認は原文、相談は整理、と役割分担。要件 §4。 |
| D4 | 整理ノート作成 | **登録時に LLM で自動生成しそのまま保存** | MVP の速度優先。確認/編集は将来。 |
| D5 | VS 登録対象 | **整理ノートのみ登録**（MVP）。原文はローカル保管・VS 非登録 | チャットは整理ノートを検索。原文は検索せずリンク閲覧（要件 §4.6）。コスト/実装を削減。 |
| D6 | ペアリング | `newsletter_id`(UUID) で原文/整理を紐づけ、台帳1レコードに集約 | 引用→原文の解決、将来の一覧/編集の土台。 |
| D7 | 引用解決 | 整理ノートの annotation `file_id` → 台帳 → 元メルマガ + 原文リンク | 出典は元メルマガ単位で表示、原文閲覧へ誘導。 |
| D8 | 時事/一般の分離 | 整理ノートに **⏱ 時事依存** と **♾ 一般原則** の2セクションを必須化 | 要件 §4.8。構造で担保。 |
| D9 | ローカル保管 | `data/newsletters.json`（索引）＋ `data/raw/*.md`＋`data/organized/*.md` | 本文をファイルで可読保管、索引は軽量。 |
| D10 | 回答スタイル | 整理ノート中心の相談相手型。出典明示・一般論は区別 | 要件 §5。 |
| D11 | ストリーミング | SDK `stream:true` を Route Handler で `ReadableStream` 転送 | 体感速度。追加依存なし。 |
| D12 | 会話状態 | クライアント保持（毎回メッセージ送信）。サーバはステートレス | MVP の単純化。`previous_response_id` は将来。 |
| D13 | 実行環境/モデル | ローカル・認証なし・`.env.local`。`OPENAI_MODEL`（既定 `gpt-4.1`） | 要件 §0。 |

## スコープ境界（再掲）

- **含む**: 登録 → 原文ローカル保存 → 整理ノート自動生成 → 整理ノートを VS 登録／整理ノート検索の回答（ストリーミング）／出典表示＋原文閲覧／期間絞り込み。
- **含まない**: 認証・公開／一覧・編集・整理ノート編集 UI／原文の VS 検索（AI 自律検索）・モード切替／鮮度注意書き・再ランキング／メール取込。

> 本資料は設計仕様であり実装コードそのものではない。コード片は意図を示す擬似コード。
