# 要件定義：メルマガ知識ベース・チャットサービス（自分用）

> 過去に受信したメールマガジンを知識ベースとして活用し、その内容に基づいて
> チャット形式で質問・相談できる、自分専用の Web サービス。
> 知識は **原文ノート（第1段階）** と **整理ノート（第2段階）** の2層で管理する。

最終更新: 2026-06-02

---

## 0. 確定した前提

| 項目 | 決定 |
|---|---|
| 技術スタック | **Next.js (App Router) + TypeScript** |
| 実行環境 | **ローカルのみ**（localhost、認証なし、API キーは環境変数） |
| 登録方法 | **テキスト貼り付け**（フォームに本文 + メタ情報をコピペ/入力） |
| 規模 | 数十件・月数件ペース（小規模） |
| 知識モデル | **2層管理**：原文ノート（忠実保存）＋ 整理ノート（抽出・整理） |
| 整理ノート作成 | **AI が登録時に自動生成しそのまま保存**（確認/編集は将来拡張） |
| Vector Store 登録 | **整理ノートのみ登録**（MVP）。原文ノートはローカル保管・VS 非登録 |
| 回答スタイル | **整理ノート中心**で相談相手型の回答。メルマガの考え方を反映 + 一般知識で補完 |
| 原文の使い分け | チャットでは原文を検索せず、**出典リンクから原文を閲覧**（事実/出典確認用） |
| 日付・出典 | **回答に出典を明示**（整理→原文へリンク）+ **期間で絞り込み検索** |
| 管理機能 | 登録 + 原文閲覧（一覧・編集・整理ノート編集は MVP 対象外） |

### 技術的な前提（裏取り済み）

- **OpenAI の Vector Store / File Search はメタデータ属性フィルタに対応**（`eq` `ne` `gt` `gte` `lt` `lte` `in` `nin`）。
  発行日を **Unix 秒（数値）属性** で登録すれば `gte`/`lte` で期間絞り込みできる。属性は最大 16 キー・各 256 文字、string/number/boolean。
- **Assistants API は 2026-08-26 終了。** 後継の **Responses API + `file_search` ツール**（`vector_store_ids` を渡す）が現行の正攻法。
- 整理ノートの自動生成は、登録時に **LLM への抽出呼び出し（1回）** で行う。

参考: OpenAI 公式ドキュメント（retrieval / tools-file-search / vector store search / deprecations / migrate-to-responses）。

## 1. このサービスで実現したいこと（ゴール）

- メルマガ群を知識ベースとし、**著者のノウハウ・判断基準・考え方・具体例を反映した回答**をチャットで得る。
- 単なる全文検索や要約に留めず、**「相談相手」として実践的に助言**してくれる。
- 知識を2層で管理することで、
  - **原文ノート**：発行時点の文脈・事実・出典を忠実に保持し、根拠確認に使う。
  - **整理ノート**：考え方・原則を抽出し、相談・方針検討・アイデア出しで応用しやすくする。
- メルマガを**少しずつ追加登録**でき、知識が育つ。
- 質問時に**発行日（時期）を考慮**できる（期間絞り込み／出典に発行日表示）。
- 時事依存の情報と、現在も応用できる考え方を**整理ノート内で区別**する。

## 2. 必要な画面（MVP）

| 画面 | 役割 |
|---|---|
| **① メルマガ登録画面** | 本文を貼り付け + メタ情報（発行日・タイトル・号数・著者・カテゴリ・タグ・発行元）を入力。登録時に整理ノートを自動生成 |
| **② チャット画面** | 質問入力・回答表示（ストリーミング）。出典（件名・発行日・著者）＋**原文を見る**リンク。任意の期間フィルタ |
| **③ 原文閲覧（モーダル）** | 出典リンクから原文ノートを表示（発行時点の文脈・本文の確認用） |

---

## 3. 必要な機能（MVP）

- **登録機能**：メタ + 本文を受け取り、(a) 原文ノートをローカル保存、(b) 整理ノートを LLM 自動生成してローカル保存、(c) 整理ノートを Vector Store に属性付きで登録。
- **整理ノート自動生成**：原文から主張・判断基準・ノウハウ・考え方・具体例・注意点・時事依存/一般原則を抽出して構造化。
- **チャット機能**：整理ノートを `file_search` で検索しつつ相談相手型で回答（ストリーミング）。
- **出典表示と原文リンク**：回答が参照した整理ノート → 元メルマガ（件名・発行日・著者）を表示し、原文ノートを開ける。
- **期間絞り込み**：発行日レンジを `file_search` の `filters` で適用。

## 4. 知識の2層管理（原文ノート / 整理ノート）

> 本サービスの中核データモデル。詳細設計は [docs/design/data-model.md](./design/data-model.md)。

### 4.1 原文ノートに保存する項目（第1段階）

元情報を忠実に残す。1メルマガ＝1原文ノート。

| 項目 | 必須 | 備考 |
|---|---|---|
| 発行日 | ✅ | `YYYY-MM-DD` |
| タイトル（件名） | ✅ | |
| 号数 | 任意 | 例「第12号」 |
| 著者 | 任意 | |
| カテゴリ | 任意 | 単一の分類 |
| タグ | 任意 | 複数可（配列） |
| 発行元 | 任意 | メルマガ名/媒体 |
| 本文 | ✅ | 貼り付けたテキストを**そのまま**保持 |

用途: 回答の根拠確認、発行時点の文脈確認。**MVP では Vector Store に登録せずローカル保管**。

### 4.2 整理ノートに保存する項目（第2段階）

原文から抽出・整理した構造化ノート。Vector Store に登録し、回答生成の土台にする。

- 要点サマリ
- 著者の主張
- 判断基準
- ノウハウ・手法
- **考え方・原則（一般化／現在も応用可）** … ♾ セクション
- 具体例
- 注意点・落とし穴
- **発行当時の状況に依存する情報（〇年時点）** … ⏱ セクション
- （補助）キーワード/関連トピック ※検索性向上

### 4.3 原文ノートから整理ノートを作成する方法

- 登録時に **LLM への抽出呼び出し（1回・非ストリーミング）** を実行。
- 抽出プロンプトで上記スキーマの構造化 Markdown を生成（[docs/design/prompt.md](./design/prompt.md) §抽出）。
- 時事依存（⏱）と一般原則（♾）を**明示セクションで分離**するよう指示。
- MVP は生成結果を**そのまま保存**（レビュー/編集は将来）。

### 4.4 原文ノートと整理ノートの関連付け方法

- 1メルマガに `newsletter_id`（UUID）を発番し、原文・整理の両方に紐づける。
- ローカル台帳は **1メルマガ＝1レコード**で、原文/整理のパスと整理ノートの `file_id` を保持。
- 整理ノートの Vector Store 属性にも `newsletter_id` を付与（引用→原文の解決に使用）。

### 4.5 Vector Store への登録方法

- **MVP では整理ノートのみ**を Vector Store に登録（自動チャンク化）。
- 属性: `note_type:"organized"`, `newsletter_id`, `issue_date`(Unix秒), `issue_date_str`, `title`, `author`, `category`, `tags`(カンマ連結), `source`, `issue_no`。
- 原文は将来「AI 自律検索」を入れる際に `note_type:"raw"` で追加登録できる設計余地を残す。

### 4.6 チャット時の使い分け

- **回答生成は整理ノートを検索**（`note_type` 実質 organized のみ）。相談・方針・アイデア出しを整理ノートで重視。
- **事実/出典確認は原文**：回答の出典として元メルマガを表示し、**「原文を見る」リンク**で原文ノートを閲覧（AI は原文を検索しない）。
- 将来拡張: AI が用途に応じて原文/整理を自律検索（function tools）／モード切替。

### 4.7 メタデータ（発行日・著者・カテゴリ・タグ等）の扱い

- 原文・整理の両ノート（整理ノートは VS 属性）に共通メタデータを保持。
- **発行日**は Unix 秒（数値）を正とし、表示用文字列も併存（期間絞り込みに使用）。
- **著者・カテゴリ**は属性として等価フィルタ（`eq`）が可能（将来の絞り込みUI余地）。
- **タグ**は属性が配列非対応のためカンマ連結文字列で保持＋本文側にも記載（厳密なタグ絞り込みは限定的、主に意味検索で活用）。

### 4.8 時事依存の情報と一般化できるノウハウの分け方

- 整理ノート内に2セクションを必ず設ける：
  - **⏱ 発行当時の状況に依存する情報**：価格・制度・仕様・時事など。可能なら「〇年時点」を明記。
  - **♾ 現在にも応用できる考え方・原則**：時期に依存しない判断基準・ノウハウ。
- 回答時、時事依存の内容には発行時期への留意を促す（プロンプトで指示）。

### 4.9 新しいメールマガジンを追加登録する手順

1. `/register` で発行日・タイトル等のメタ情報と本文を入力（貼り付け）。
2. 登録実行 → サーバが `newsletter_id` 発番 → 原文ノートをローカル保存。
3. LLM が整理ノートを自動生成 → ローカル保存。
4. 整理ノートを Vector Store に属性付きで登録（完了までポーリング）。
5. 台帳に1レコード追記。以降は同手順で追加。重複は本文ハッシュで検知。

## 5. Chatbot の回答方針

- **役割**：整理ノート群の考え方・判断基準を内面化した「相談相手」。
- **動作**：整理ノートを `file_search` で参照し、考え方・ノウハウを反映して助言。一般知識で補完するが、メルマガ由来か一般論かを区別。
- **出典**：参照した整理ノートに対応する元メルマガ（件名・発行日・著者）を提示し、原文へリンク。
- **時期**：時事依存の内容はいつ時点の情報かに留意して述べる。
- **誠実さ**：該当情報が無ければ断定しない。
- 詳細は [docs/design/prompt.md](./design/prompt.md)。

## 6. 発行日時・出典情報の扱い（今回のスコープ）

- **出典明示 + 原文リンク**：整理ノートの引用（`file_id`）→ 台帳 → 元メルマガ（件名・発行日・著者）を表示し、原文ノートを開ける。
- **期間絞り込み**：`issue_date`（数値）に `gte`/`lte`。
- **今回は対象外（将来拡張）**：「〇年時点の情報です」の鮮度注意書きの自動付与／新しさ優先の再ランキング。

## 7. MVP の範囲（最小構成）

**含む:**
- 登録（メタ + 本文貼り付け）
- 原文ノートのローカル保存
- 整理ノートの **AI 自動生成** + ローカル保存
- 整理ノートの Vector Store 登録（属性付与）
- 整理ノート検索による回答（ストリーミング）
- 出典表示（件名・発行日・著者）＋ **原文を見る**（原文閲覧モーダル）
- 期間絞り込み

**含まない:**
- 認証・公開・マルチユーザー
- 一覧・編集・削除 UI、整理ノートの確認/編集/再生成 UI
- 原文ノートの Vector Store 検索（AI 自律検索）・相談/事実確認モード切替
- 鮮度注意書き・新しさ優先の再ランキング
- メール転送・ファイル取込

### 想定アーキテクチャ（MVP）

```
[ブラウザ]
  ├─ 登録フォーム ─▶ /api/ingest
  │     ├─ 原文ノート保存（ローカル data/raw）
  │     ├─ 整理ノート生成（LLM）→ 保存（data/organized）
  │     └─ 整理ノートを Vector Store 登録（属性付与）
  ├─ チャットUI ──▶ /api/chat ─▶ Responses API + file_search(整理ノート, 期間filters)
  └─ 原文を見る ──▶ /api/newsletters/{id}/raw（ローカル原文を返す）

シークレット/VS ID は .env.local（ローカル専用・認証なし）
```

## 8. 将来的に拡張できる点

- **整理ノートの品質運用**：生成結果のレビュー/編集 UI、再生成、版管理。
- **原文の活用強化**：原文ノートも Vector Store に登録し、AI が用途に応じて原文/整理を**自律検索**（function tools）。相談/事実確認モード切替。
- **管理機能**：一覧・検索・編集・削除・再同期。著者/カテゴリ/タグでの絞り込み UI。
- **取込手段**：ファイルアップロード（.eml/.pdf/.md）→ メール転送による自動取込。
- **日付の高度化**：鮮度注意書き、発行日による再ランキング。
- **公開/マルチユーザー化**：認証・権限（Supabase 等）、デプロイ（Vercel）。
